Fixed V1 Route

IM V1 路線與必要模組推導

本頁說明：在已確定的 V1 route 下，如何從普通用戶與商家 / 商業主的核心需求出發，判斷哪些模組是 V1 必要、哪些只需要輕量佔位、哪些應放到後續擴展，最後組成第一版 IM 架構。

角色需求 必要模組分類 參考流程選擇 範圍切割
推導原則

不是把所有模組都做完

V1 只做能支撐優先角色需求與核心 route 的必要模組。

AI、CRM、Bot/API、mini-apps 與完整商務是後續擴展方向，不是 V1 架構證明。

Priority Role Needs

先看優先角色，不先展開所有模組

依據 compare_features 的角色需求映射，本頁只用普通用戶與商家 / 商業主作為 V1 判斷起點。

Evidence-backed

普通用戶

  • 穩定聊天
  • 好友邀請
  • 群組
  • 狀態更新
  • 安全封鎖

V1 解讀： 必須支撐穩定個人聊天、邀請入口、群組參與、基本狀態 / 更新表面，以及封鎖 / 檢舉安全動作。

Evidence-backed

商家 / 商業主

  • 商業檔案
  • 頻道經營
  • 商品卡
  • AI 客服
  • 外部投放

V1 解讀： 必須支撐商業身份、頻道經營、聊天 / 頻道內的輕量商品卡，以及外部流量承接。

AI 客服不代表 V1 要做完整 AI support system；這裡只保留後續服務接入 hook。

Module Necessity Classification

先分類，再決定哪些模組值得詳細 mock

被 compare_features 映射到角色需求的模組，不會自動進入 V1 詳細架構。

Group A

V1 必要，需詳細推導與 mock

這些模組直接決定固定 route 是否成立，必須有 competitor reference flow、adopted / rejected pattern 與架構含義。

  • messaging-identity
  • external-entry
  • channel-loop
  • community-governance
  • trust-risk
Group B / C

V1 必要但輕量 / 輕量佔位

需要支撐優先角色，但不能擴成完整 commerce、feed 或 stories 產品。

business-commerce business profile、business chat label、product card object；不做 marketplace、payment、order fulfillment。
status-updates lightweight update/status object，可回到 inbox / chat；不做完整 stories editor、feed 或 recommendation。
Group D

Later extension，不進 V1 詳細 mock

這些是重要後續能力，但不是證明 V1 route 的必要條件。

  • support-crm / full CRM
  • platform-ai-ops / full AI agent
  • mini-apps / bot/API ecosystem
  • b2b-selling / advanced recommendation
  • full marketplace / payment / fulfillment

Necessary Module → Reference Flow Choice

只對必要模組做詳細競品流程比較

流程比較用來決定 adopted / rejected pattern；Later 模組只保留 hook，不做完整 mock。

Module V1 必要原因 參考產品 採用 pattern 不採用 pattern 架構含義 流程比較
messaging-identity 普通用戶的穩定聊天與商家身份標籤都依賴它。 WhatsApp、LINE、Telegram、WeChat 統一聊天列表 + 清楚類型標籤 + 個人 / 商業身份邊界。 把商業流程寫死在個人聊天模型；或直接複製超級平台。 形成 Identity Layer 與 Inbox / Relationship Layer。 開啟
external-entry 好友邀請、外部投放、頻道入口都需要可追蹤入口。 LINE QR / ID、Telegram public link、WhatsApp invite / click-to-chat、TikTok / Instagram CTA 入口模型記錄目標類型、來源、加入規則與風險狀態。 所有入口都直接丟進完整群聊。 形成 External Entry 與 Public Channel 的前段承接。 開啟
channel-loop 商家頻道經營與公域內容進討論的核心。 Telegram Channel、WhatsApp Channels、Discord announcement + forum、LINE OA 低噪音頻道 + 明確討論 CTA + 可分享公開內容。 純廣播無討論；或頻道內完整留言噪音流。 Public Channel 接 Community Group / Topic Layer。 開啟
community-governance 普通用戶群組與主題互動需要可找、可管、可回訪。 Telegram public group + topics、Discord onboarding / forum、LINE OpenChat、Facebook Groups 主題列表優先，群組首頁承接成員狀態與規則。 直接進單一大群訊息流；或 Discord-level server system。 形成 Community Group + Topic Layer。 開啟
trust-risk 安全封鎖、公開社群與商業訊息都需要最低信任底座。 Discord reports / AutoMod、WhatsApp Business quality、LINE block / rate limits admin、report、block、mute、remove member、basic moderation log。 Day one 無治理；或直接做企業級合規後台。 Governance Layer 橫跨公開、社群、收件箱。 開啟
business-commerce 商家 / 商業主是優先角色，但 V1 只需輕量商務表示。 WhatsApp Business、LINE OA、WeChat OA / Mini Program、Shopify Inbox / Instagram 商業檔案、商業聊天標籤、商品卡物件，可分享進聊天 / 頻道。 完整 marketplace、payment、order / fulfillment system。 Business Lightweight Layer，不佔據 IM 主流程。 開啟
status-updates 普通用戶需要近況 / 更新，但不應成為大內容產品。 WhatsApp Status / Updates、Instagram Stories、Snapchat Stories 輕量 update/status object，可回覆或回到 inbox / chat。 完整 stories editor、社交 feed、推薦系統。 Status / Updates Surface，只作輕量入口。 輕量佔位

Later Expansion Notes

保留 hook，但不提前建平台

這些方向可以在資料模型上留接點，但不應消耗 V1 的互動 mock 與架構證明。

AI support

只作 support / automation hook；不做 V1 AI 客服系統。

CRM

只保留商業聊天標籤與未來接入點；不做 ticketing / routing / SLA。

Bot/API

只預留授權與風控概念；不做完整 bot ecosystem。

Mini-apps

只作未來服務容器方向；不做 marketplace 或開放平台。

Full commerce

商品卡留在聊天 / 頻道；payment、order、fulfillment 後置。
