IM V1 產品路線決策
決策總覽

建議核准：內容導向的社群 IM 作為 V1 路線

基於現有競品比較、產品架構與互動流程研究，V1 不應嘗試同時複製 Telegram、Discord、Reddit、LINE 與 WhatsApp。建議先核准一條可寫 PRD 的路線：公開內容進站、頻道建立信任、主題化討論承接互動、收件箱沉澱關係。

老闆本次要決定

Approve / Adjust / Reject

預設建議是 approve 這條 V1 路線，並只調整五個架構預設：頻道到討論、大群第一屏、收件箱組織、公開預覽邊界、day-one 治理。

會後產出 若方向被核准，下一步可直接寫 PRD 與 MVP 驗證計畫。
外部內容 / 邀請 公開頻道預覽 關注 / 加入 結構化討論 主題互動 收件箱 / 私聊留存
推薦

V1 做「內容導向的社群 IM」，而不是大而全的社群平台或純私訊工具。

因為

既有架構已收斂到「外部入口 → 頻道 → 主題討論 → 收件箱」；這條路能同時解決進站、互動與回訪。

證據

參考 `product_architecture/index.html` 的北極星路徑、`compare_features/index.html` 的 27 款產品能力分布、`interaction_playground/app.js` 的 channel-content / group-topics 流程。

風險

主題結構若太早過重，會增加加入摩擦；若太輕，公開社群會被訊息流噪音稀釋。

假設

使用者願意先預覽頻道內容，再因明確討論入口加入社群。此假設需 MVP 驗證。

什麼會改變建議

若測試顯示使用者主要需求是純私聊或純內容瀏覽，V1 應縮回 WhatsApp-like 私訊或頻道-only 路線。

V1 核心閉環

推薦預設：只做能讓閉環端到端成立的模組

判斷原則：模組若不能支撐「公開入口 → 討論 → 關係留存 → 回訪」，就不進 V1，避免第一版變成 Telegram + Discord + Reddit + LINE 的混合體。

01

外部入口

使用者動作
從內容、邀請連結、搜尋結果或 QR Code 進入。
產品回應
打開公開頁，不要求立即登入。
必要模組
公開連結、基礎身份邊界、搜尋 / 發現。
證據 / 推理
`product_architecture/index.html` 已把外部內容引流放在北極星路徑第一步。
風險
公開入口若內容不足，會變成空頁或低信任入口。
02

公開頻道預覽

使用者動作
加入前先看頻道定位與公開貼文。
產品回應
呈現頻道 profile、精選貼文與加入 / 討論 CTA。
必要模組
頻道、公開頁、身份、信任。
證據 / 推理
`reports/內容頻道閉環.md` 與互動 Demo 都把頻道視為內容承接面。
風險
預覽太少無法建立信任；預覽太多會增加治理成本。
03

關注 / 加入

使用者動作
先關注頻道，或加入對應社群。
產品回應
把一次性瀏覽轉成可通知、可回訪的關係。
必要模組
帳號、成員狀態、通知預設。
證據 / 推理
`compare_features/index.html` 顯示主流 IM / 頻道產品都依賴 follow、join 或聊天列表沉澱關係。
風險
若加入流程太重，會折損公開入口帶來的低摩擦。
04

討論入口

使用者動作
從內容 CTA 進入討論，而不是直接掉進完整大群訊息流。
產品回應
預設帶到主題列表或指定主題。
必要模組
群組、主題、公告、基本規則。
證據 / 推理
互動 Demo 中 Telegram、Discord / Forum-like 流程都呈現內容到討論的結構化承接。
風險
主題分類若不準，會讓新使用者不知道該去哪裡。
05

主題式互動

使用者動作
選擇問答、公告、活動、資源等主題參與。
產品回應
在主題內顯示訊息流、置頂、管理與檢舉入口。
必要模組
主題訊息、管理員、檢舉 / 封鎖 / 靜音。
證據 / 推理
`interaction_playground/app.js` 的 group-topics 情境把 Topics 作為大群降噪方式。
風險
過度結構化會像 forum；過少結構會像失控群聊。
06

收件箱 / 私人關係

使用者動作
回到收件箱看私聊、群組、頻道更新與通知。
產品回應
用類型標籤、未讀與篩選維持秩序。
必要模組
統一收件箱、私聊、未讀、通知、篩選。
證據 / 推理
`product_architecture/index.html` 已把收件箱定位為私聊、群組、頻道與商業對話的承接底座。
風險
頻道更新若壓過私人對話，會破壞 IM 心智。
07

回訪

使用者動作
因私人回覆、主題更新或頻道更新再次回來。
產品回應
通知帶回正確頻道、群組主題或私聊。
必要模組
通知設定、深連結、訊息狀態。
證據 / 推理
IM 產品的留存心智來自聊天列表與通知，而不是單次內容瀏覽。
風險
通知策略過強會打擾；過弱會讓社群無法回訪。
模組對閉環地圖

推薦切法：核心閉環先做，支援模組只做到可用

V1 的模組優先級不是功能重要性排序，而是閉環必要性排序。能讓使用者完成閉環的是 V1；能優化營運或商業化的是 Later。

核心閉環模組

必須進 V1，否則產品路線不成立

身份

因為公開預覽、加入、私聊與治理都需要基本身份邊界。

頻道

因為它是外部內容進站後的信任與內容承接面。

群組 / 主題

因為公開內容必須轉成可參與、可管理的多人互動。

收件箱

因為 IM 的留存場域是聊天列表、未讀、私聊與回訪。

支援模組
搜尋 / 發現

V1 只做能找到頻道、群組與聊天的基本能力；推薦與 SEO 先不做。

信任 / 治理

V1 只做公開社群最低可用治理：管理員、檢舉、封鎖、靜音、移除。

通知

V1 只保證私聊、群組 / 主題、頻道更新能回到正確位置。

後續 / 擴展
創作者工具

進階排程、分析、多管理員放 Later；V1 先證明頻道到討論的閉環。

錢包 / 積分

激勵層會放大複雜度；除非核心社群閉環已被證明，否則不進 V1。

API / Bot

平台能力先保留方向，不作為第一版可交付範圍。

進階分析

不要用假儀表板製造確定性；V1 只需要基本事件紀錄支援驗證。

關鍵架構決策

不是列問題，而是給出可核准的推薦預設

以下五張卡片是 PRD 前必須對齊的架構預設。老闆不需要從零解題，只需要 approve、adjust 或 reject。

決策 01 開啟互動 Demo

頻道內容如何導向討論？

推薦預設：混合路徑：公開頻道預覽 → 討論 CTA → 主題優先群組。

為什麼：單純留言容易變成內容產品；單純跳群容易掉進噪音。混合路徑能保留低摩擦入口與結構化討論。

證據：`interaction_playground/app.js` 的 channel-content 情境、`reports/內容頻道閉環.md`、`product_architecture/index.html` 的北極星路徑。

不採用替代：不採純原生留言；不採純另開群；不先做完整 forum。

風險：結構增加可能提高加入摩擦。

假設：使用者看完頻道內容後，願意透過明確 CTA 進入主題討論。

老闆動作：Approve 混合路徑；或調整頻道貼文是否允許有限留言預覽。

決策 02 開啟互動 Demo

進入大型群組後第一屏看什麼？

推薦預設：主題列表優先，並露出輕量公告 / 規則。

為什麼：新使用者最怕不知道去哪裡；第一屏如果是完整訊息流，噪音會掩蓋社群結構。

證據：互動 Demo 的 Discord Forum / Telegram Topics 參考，以及 `product_architecture/index.html` 的「群組與主題分區」。

不採用替代：不採完整訊息流 first；不採重 onboarding first。

風險：初版 topic taxonomy 可能錯。

假設：營運可以提供 3-6 個預設主題模板，讓社群冷啟動不空轉。

老闆動作：Approve 主題列表 first；或指定 V1 首批標準主題模板。

決策 03

收件箱如何組織私聊、群組與頻道？

推薦預設：統一收件箱 + 清楚類型標籤 + 篩選。

為什麼：IM 的心智在收件箱；完全分開會漏看，完全混合會變吵。

證據：`compare_features/index.html` 顯示 WhatsApp、LINE、Telegram 都以聊天列表維持關係入口；`product_architecture/index.html` 把收件箱定位為通訊核心。

不採用替代：不採純時間序；不先做 priority inbox 或 AI summary。

風險：未讀與通知邏輯若不清楚，頻道更新會壓過私人聊天。

假設：類型標籤與篩選足以處理 V1 的訊息秩序。

老闆動作：Approve 統一收件箱；或要求 V1 預設分頁而非篩選。

決策 04

登入 / 加入前公開什麼？

推薦預設：公開頻道 profile + 精選公開貼文；討論需登入 / 加入。

為什麼：公開預覽降低摩擦，但討論內容公開會擴大隱私與治理成本。

證據：`product_architecture/index.html` 把公開頻道預覽列為核心入口；`reports/平台安全治理.md` 指向公開社群需要清楚治理邊界。

不採用替代：不採無預覽；不採完整討論公開。

風險：公開內容規則若不清楚，會造成索引、分享與撤回爭議。

假設：精選公開貼文足以建立加入前信任。

老闆動作：Approve 公開邊界；或指定是否允許少量討論摘要公開。

決策 05

Day one 需要哪些治理？

推薦預設：管理員 + 檢舉 / 封鎖 / 靜音 + 移除成員 + 主題層級 moderation 基礎。

為什麼：公開社群 IM 如果沒有基本安全與控制邊界，很難被營運與使用者信任。

證據：`reports/平台安全治理.md` 與 `product_architecture/index.html` 都把信任治理視為底座；`compare_features/index.html` 顯示 Discord、Telegram 等社群產品在治理上具參考價值。

不採用替代：不採 minimal admin only；不先做完整 trust / safety system。

風險：治理過重會拖慢 V1；治理過輕會讓公開社群不可用。

假設：V1 的社群規模仍可由人工管理與基本工具承接。

老闆動作：Approve 基本治理；或指定哪些治理能力可延後。

剩餘假設

只保留會影響 V1 路線的假設

這些不是要老闆現場全部解完，而是核准推薦路線後，PRD 與 MVP 驗證必須追的風險。

假設 01

公開頻道預覽足以提高加入意願

為什麼重要：如果預覽不能建立信任，content-led 入口就不成立。

如何驗證：用 2-3 個真實內容頻道測試公開頁 → join / follow CTA。

若為假：V1 應縮小公開預覽，改強化邀請與私域導入。

假設 02

主題列表能降低大群進入成本

為什麼重要：這決定 V1 是否要在群組第一屏預設 Topics。

如何驗證：比較「訊息流 first」與「主題 first」的首次回覆與停留行為。

若為假：改為公告 + 熱門訊息流 first，主題退到第二層。

假設 03

統一收件箱不會被頻道更新污染

為什麼重要：若收件箱太吵，IM 關係留存會被破壞。

如何驗證：在 MVP 中測試類型標籤、篩選與預設靜音策略。

若為假：V1 改成 Chats / Channels 分頁，或頻道更新預設進 Updates。

假設 04

基本治理足以支撐第一批公開社群

為什麼重要：治理不足會直接影響可用性與品牌風險。

如何驗證：先用小規模社群測試檢舉、封鎖、移除與 topic moderation。

若為假：提前加入角色權限或更嚴格的加入審核。
