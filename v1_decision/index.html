IM V1 架構與必要模組

從普通用戶與商家 / 商業主的核心需求出發，收斂成以「身份 + 收件箱」為底座，支撐私聊 / 群聊、公開內容 / 社群、輕量商業三類旅程的 IM 架構。

角色需求 模組地圖 決策依據 情境模擬 範圍切割

三類核心旅程

  1. 私聊 / 群聊旅程普通用戶建立身份、邀請好友，進入私聊或群聊，並用收件箱留存關係。
  2. 公開內容 / 社群旅程外部內容或邀請先進入公開頻道，再依情境導向關注、加入、群組、討論串或主題。
  3. 商業輕量旅程商家建立商業身份、經營頻道、分享商品卡，並把高意圖用戶帶回聊天。

AI 輔助、CRM、Bot/API、mini-apps、完整商務後置。

需求

優先角色需求

C 端

普通用戶

  • 穩定聊天
  • 好友邀請
  • 群組
  • 狀態更新
  • 安全封鎖

V1： 必須支撐穩定個人聊天、邀請入口、群組參與、基本狀態 / 更新，以及封鎖 / 檢舉安全動作。

B 端

商家 / 商業主

  • 商業檔案
  • 頻道經營
  • 商品卡
  • AI 客服
  • 外部投放

V1： 必須支撐商業身份、頻道經營、聊天 / 頻道內的輕量商品卡，以及外部流量承接。

AI 客服不代表 V1 要做完整 AI 客服系統；這裡只保留後續服務接入點。

聊天底座身份 + 收件箱作為共享入口。把私聊、群組、頻道、商業訊息拆成多個主入口。Evidence-backedmessaging-identity 成為 V1 底座，所有回訪先回到收件箱。

用情境證明 V1 決策

這裡不列完整情境清單，只說每組 mock 為什麼重要、證明哪個 V1 判斷。

模組 為什麼重要 打開情境 情境證明什麼
messaging-identity 如果入口分裂，使用者會分不清聊天、頻道和商業關係。
混合收件箱私聊
 身份 + 收件箱可以用類型標籤承接多種關係，不需要多個主入口。
external-entry 外部連結同時是增長入口和風險邊界。
好友邀請群組邀請商業 CTA
 入口先判斷目標類型、來源、加入規則，再進入聊天、群組、頻道或商業檔案。
channel-loop 公開內容需要低壓力預覽，也需要回到可治理討論。
公開頻道貼文討論
 頻道不是動態流；它是公開預覽、關注和討論 CTA 的閉環。
community-governance 小群和大型社群不能被同一個主題優先模型綁死。
小群直聊主題模式
 小群直接聊天；主題是大型或高流量社群的模式，不是所有群的預設路徑。
trust-risk 公開入口、商業訊息和社群互動都需要基本安全底線。
用戶安全加入風險
 V1 需要封鎖 / 檢舉 / 靜音、加入限制和基本管理紀錄，但不做完整合規平台。
business-commerce 商業需求要被承接，但不能把 V1 拉成交易平台。
商業檔案聊天商品卡
 商業身份和商品卡足以保留購買意圖；支付、訂單、履約後置。
status-updates 近況更新可以增加回訪，但不應變成內容產品。
個人狀態回訪路徑
 狀態是輕量更新物件；互動回到收件箱 / 聊天，不做動態流或推薦。

  • 小群直接聊天

  • 用戶封鎖 / 檢舉 / 靜音

  • 管理員移除 / 禁言成員
  • 商業訊息可封鎖 / 退訂

Business commerce
  • 商家建立商業檔案
  • 用戶在聊天中看到商品卡
  • 商品卡出現在頻道或私聊
  • 高意圖用戶回到商業聊天
型錄同步、訂單摘要接點、服務整合。完整市集、支付、訂單 / 履約。

  • 他人回覆進私聊
  • 商家 / 頻道發布輕量更新
更豐富模板、商業活動狀態。完整限時動態編輯器、社交動態流、推薦。
後續擴展說明

保留接點，但不提前建平台

這些方向可以在資料模型上留接點，但不應消耗 V1 的互動 mock 與架構證明。

AI 客服

只作客服 / 自動化接點；不做 V1 AI 客服系統。

CRM

只保留商業聊天標籤與未來接入點；不做工單 / 分流 / SLA。

Bot/API

只預留授權與風控概念；不做完整 Bot 生態。

Mini-apps

只作未來服務容器方向；不做市集或開放平台。

完整商務

商品卡留在聊天 / 頻道；支付、訂單、履約後置。
